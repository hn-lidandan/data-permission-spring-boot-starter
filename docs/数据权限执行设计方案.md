# 数据权限执行设计方案

## 1. 文档目标

本文档整理当前阶段 SDK 在业务服务侧执行数据权限的设计方案，覆盖三类能力：

- SQL 行级权限过滤
- 字段脱敏
- 导出控制

当前阶段重点是先完成 MVP，验证真实业务请求链路中 SDK 能基于权限上下文完成本地权限执行。权限上下文由权限服务返回，并已通过 Web 拦截器绑定到当前请求线程。

## 2. 总体约定

SDK 执行权限时统一从 `PermissionContextHolder` 读取当前用户权限上下文，不在 SQL 拦截器、字段脱敏模块、导出模块里重复调用权限服务。

当前权限上下文示例：

```json
{
  "tenant_id": "ddd111",
  "user_id": "dfew123",
  "deny_all": false,
  "dept_scope": ["d001", "d002"],
  "asset_group_scope": ["ag001", "ag002"],
  "data_level_scope": ["public", "internal"],
  "resource_object": {
    "doris": ["security_log", "alert_event"],
    "postgre_sql": ["asset", "report"]
  },
  "field_policies": [
    {
      "field_name": "phone",
      "mask_regex": "PHONE",
      "permission_type": "MASK",
      "scene": ["DETAIL"]
    },
    {
      "field_name": "raw_log",
      "mask_regex": null,
      "permission_type": "HIDE",
      "scene": ["DETAIL", "EXPORT"]
    }
  ],
  "export_policies": [
    {
      "is_allowed": true,
      "max_rows": 5000,
      "need_approval": false,
      "resource_id": "security_log",
      "resource_type": "doris"
    }
  ]
}
```

当前阶段约定：

- `deny_all=true` 时默认拒绝。
- 权限上下文不存在或已过期时默认拒绝。
- `resource_id` 必须等于 SQL 物理表名。
- SQL 表名不在 `resource_object` 中时，认为 SDK 当前不管理该表，直接放行。
- SQL 表名在 `resource_object` 中时，追加 SQL 行级过滤条件。

## 3. 场景设计

SDK 需要感知当前请求场景，后续 SQL 过滤、字段脱敏、导出控制都从统一场景中读取。

当前支持的场景：

| 场景 | 说明 |
| --- | --- |
| `QUERY` | 普通列表查询 |
| `DETAIL` | 详情查看 |
| `EXPORT` | 数据导出 |

默认场景：

```text
QUERY
```

后续可以通过注解或 API 设置当前场景，例如：

```java
@PermissionScene("DETAIL")
```

或者：

```java
@PermissionScene("EXPORT")
```

建议新增 `DataPermissionSceneHolder`，用 `ThreadLocal` 保存当前请求场景。请求结束后需要清理，避免线程池复用导致场景串扰。

各模块使用场景的方式：

| 模块 | 当前阶段场景规则 |
| --- | --- |
| SQL 行级过滤 | `tenant_id`、`dept_id` 对所有场景都生效 |
| 字段脱敏 | 根据 `scene + field_name` 匹配字段策略 |
| 导出控制 | 只在 `EXPORT` 场景执行导出策略校验 |

## 4. SQL 行级权限过滤

### 4.1 目标

SQL 行级过滤用于控制当前用户能查询哪些数据行。

当前阶段只过滤两个字段：

```text
tenant_id
dept_id
```

二者是 `AND` 关系：

```sql
tenant_id = ?
AND dept_id IN (?, ?)
```

### 4.2 资源识别规则

当前阶段不配置资源映射，直接使用权限上下文中的 `resource_object` 判断。

约定：

```text
权限上下文里的 resource_id 必须等于 SQL 物理表名
```

例如：

```json
"resource_object": {
  "doris": ["security_log"],
  "postgre_sql": ["asset"]
}
```

表示 SDK 当前只对以下表做 SQL 权限过滤：

```text
security_log
asset
```

表名匹配规则：

- `SELECT * FROM security_log` 提取表名 `security_log`。
- `SELECT * FROM security_log t` 提取表名 `security_log`。
- `SELECT * FROM public.security_log` 当前阶段提取最终表名 `security_log`。

### 4.3 放行与过滤规则

SQL 表名不在 `resource_object` 中：

```text
直接放行，不改写 SQL。
```

SQL 表名在 `resource_object` 中：

```text
追加 tenant_id AND dept_id 过滤条件。
```

这个阶段 `resource_object` 的含义是：

```text
当前用户上下文中需要 SDK 做 SQL 权限过滤的资源集合。
```

注意：当前阶段 SDK 无法区分“非受控表”和“受控但当前用户无权限的表”。如果后续要严格拒绝无权限受控表，需要额外引入受控资源清单。

### 4.4 改写示例

原 SQL：

```sql
SELECT * FROM security_log
```

权限上下文：

```json
{
  "tenant_id": "ddd111",
  "dept_scope": ["d001", "d002"]
}
```

改写后：

```sql
SELECT * FROM security_log
WHERE tenant_id = ?
  AND dept_id IN (?, ?)
```

如果原 SQL 已有业务条件：

```sql
SELECT * FROM security_log WHERE status = ?
```

改写后：

```sql
SELECT * FROM security_log
WHERE status = ?
  AND tenant_id = ?
  AND dept_id IN (?, ?)
```

参数追加顺序：

```text
原业务参数
tenant_id
dept_scope...
```

### 4.5 异常规则

如果 SQL 表名命中 `resource_object`，但出现以下情况，默认拒绝：

- `PermissionContext` 不存在。
- `deny_all=true`。
- 上下文已过期。
- `tenant_id` 为空。
- `dept_scope` 为空。
- SQL 结构不支持安全改写。

当前阶段建议直接抛权限异常，不改写成 `WHERE 1 = 0`，便于真实业务服务接入时定位问题。

### 4.6 SQL 支持范围

当前阶段支持：

- 单表 `SELECT`
- 单表 `COUNT`
- 原 SQL 已有 `WHERE` 时追加 `AND`
- 原 SQL 无 `WHERE` 时新增 `WHERE`

当前阶段暂不支持：

- `JOIN`
- `UNION`
- 子查询
- `WITH / CTE`
- `INSERT`
- `UPDATE`
- `DELETE`
- 复杂聚合查询

处理规则：

```text
表名不在 resource_object 中：直接放行。
表名在 resource_object 中，但 SQL 结构不支持：默认拒绝。
```

## 5. 字段脱敏

### 5.1 目标

字段脱敏用于控制接口返回字段的展示方式。它不参与 SQL 改写，只在响应数据返回阶段生效。

SQL 行级过滤控制：

```text
能查哪些行。
```

字段脱敏控制：

```text
字段以明文、脱敏、隐藏还是拒绝的方式返回。
```

### 5.2 策略来源

字段策略来自 `PermissionContext.fieldPolicies`。

示例：

```json
[
  {
    "field_name": "phone",
    "mask_regex": "PHONE",
    "permission_type": "MASK",
    "scene": ["DETAIL"]
  },
  {
    "field_name": "raw_log",
    "mask_regex": null,
    "permission_type": "HIDE",
    "scene": ["DETAIL", "EXPORT"]
  }
]
```

### 5.3 匹配规则

当前阶段先按以下维度匹配：

```text
field_name + scene
```

当前阶段不强制匹配 `resource_id`，因为现有 `field_policies` 里没有资源维度。

匹配规则：

- 当前 `scene` 命中策略的 `scene` 列表时，策略生效。
- 策略 `scene` 为空时，按 `ALL` 处理。
- 没有匹配到策略时，默认 `SHOW`。

### 5.4 字段动作

| `permission_type` | SDK 行为 |
| --- | --- |
| `SHOW` | 原样返回 |
| `MASK` | 按 `mask_regex` 脱敏后返回 |
| `HIDE` | 返回 `null` |
| `DENY` | 拒绝本次响应，抛权限异常 |

示例：

当前场景为 `DETAIL`，原始响应：

```json
{
  "phone": "13812345678",
  "raw_log": "raw log content",
  "username": "zhangsan"
}
```

字段策略：

```text
phone -> MASK -> DETAIL
raw_log -> HIDE -> DETAIL, EXPORT
```

处理后：

```json
{
  "phone": "138****5678",
  "raw_log": null,
  "username": "zhangsan"
}
```

### 5.5 后续演进

当前 `field_policies` 不带 `resource_id`，同名字段会按全局字段策略处理。后续正式版本建议字段策略增加资源维度：

```json
{
  "resource_type": "doris",
  "resource_id": "security_log",
  "field_name": "raw_log",
  "permission_type": "HIDE",
  "scene": ["DETAIL", "EXPORT"]
}
```

最终匹配规则升级为：

```text
resource_id + scene + field_name
```

## 6. 导出控制

### 6.1 目标

导出控制用于判断当前用户是否允许导出某个资源，以及导出数据量是否超过限制。

导出控制只在 `EXPORT` 场景生效。

### 6.2 策略来源

导出策略来自 `PermissionContext.exportPolicies`。

示例：

```json
[
  {
    "is_allowed": true,
    "max_rows": 5000,
    "need_approval": false,
    "resource_id": "security_log",
    "resource_type": "doris"
  }
]
```

### 6.3 匹配规则

当前阶段按以下维度匹配：

```text
resource_id + scene=EXPORT
```

当前阶段仍然约定：

```text
resource_id 等于 SQL 物理表名。
```

### 6.4 校验流程

导出前执行以下校验：

1. 判断当前是否为 `EXPORT` 场景。
2. 根据 `resource_id` 查找 `export_policies`。
3. 找不到策略时默认拒绝。
4. `is_allowed=false` 时拒绝。
5. `need_approval=true` 时当前阶段拒绝。
6. `estimatedRows > max_rows` 时拒绝。
7. 校验通过后允许导出。

当前阶段不建议 SDK 自动把任意查询 SQL 改写为 `COUNT` SQL。导出预估条数建议由业务服务显式传入。

示例 API：

```java
DataPermissionExport.check("security_log", estimatedRows);
```

后续可以扩展为注解：

```java
@PermissionExport(resourceId = "security_log")
```

## 7. 建议模块划分

### 7.1 场景上下文

```text
context/
  DataPermissionSceneHolder
```

职责：

- 保存当前请求场景。
- 默认返回 `QUERY`。
- 请求结束时清理场景。

### 7.2 SQL 过滤模块

```text
sql/
  SqlTableExtractor
  PermissionResourceMatcher
  PermissionFilterBuilder
  SqlRewriteEngine
  DataPermissionMyBatisInterceptor
```

职责：

- `SqlTableExtractor`：使用 JSqlParser 提取单表物理表名。
- `PermissionResourceMatcher`：判断表名是否存在于 `PermissionContext.resourceObject`。
- `PermissionFilterBuilder`：根据 `tenant_id`、`dept_scope` 构建过滤条件和参数。
- `SqlRewriteEngine`：使用 SQL AST 改写 SQL。
- `DataPermissionMyBatisInterceptor`：拦截 MyBatis SQL，替换 SQL 并追加参数。

### 7.3 字段脱敏模块

```text
masking/
  PermissionField
  DataMaskingEngine
  PermissionJacksonModule
```

职责：

- 根据 `scene + field_name` 匹配字段策略。
- 执行 `SHOW / MASK / HIDE / DENY`。
- 接入 Jackson 序列化流程。

### 7.4 导出控制模块

```text
export/
  DataPermissionExport
  ExportPolicyMatcher
  ExportGuard
```

职责：

- 根据 `resource_id` 匹配导出策略。
- 校验 `is_allowed`、`max_rows`、`need_approval`。
- 为后续 AOP 或注解式导出拦截预留扩展点。

## 8. 整体执行流程

```text
请求进入
  -> Web 拦截器获取 PermissionContext
  -> PermissionContextHolder 绑定当前上下文
  -> DataPermissionSceneHolder 默认设置 QUERY

业务执行
  -> 详情接口设置 scene = DETAIL
  -> 导出接口设置 scene = EXPORT

SQL 执行前
  -> MyBatis 拦截器获取原 SQL
  -> JSqlParser 解析 SQL
  -> 提取单表物理表名
  -> 判断表名是否在 resource_object 中

     不在：
       原 SQL 放行，不改写

     在：
       校验 tenant_id、dept_scope
       追加 tenant_id AND dept_id 条件
       追加 MyBatis 参数
       执行改写后的 SQL

响应返回前
  -> 字段脱敏模块读取 field_policies
  -> 根据 scene + field_name 执行 SHOW / MASK / HIDE / DENY

导出前
  -> 导出模块读取 export_policies
  -> 根据 resource_id 校验导出权限
  -> 校验 max_rows 和 need_approval
  -> 通过后继续导出

请求结束
  -> 清理 PermissionContextHolder
  -> 清理 DataPermissionSceneHolder
```

## 9. 当前 MVP 结论

SQL 行级过滤：

- 不依赖配置。
- `resource_id` 等于 SQL 物理表名。
- 表名不在 `resource_object` 中直接放行。
- 表名在 `resource_object` 中才追加权限条件。
- 暂时只过滤 `tenant_id`、`dept_id`。
- `tenant_id` 和 `dept_id` 是 `AND` 关系。
- 只支持单表 `SELECT` 和单表 `COUNT`。
- 命中 `resource_object` 但 SQL 不支持时默认拒绝。

字段脱敏：

- 按 `field_name + scene` 匹配。
- 支持 `SHOW / MASK / HIDE / DENY`。
- 未匹配策略时默认 `SHOW`。
- 当前阶段暂不强制匹配 `resource_id`。

导出控制：

- 只在 `EXPORT` 场景生效。
- 按 `resource_id` 匹配 `export_policies`。
- 找不到导出策略默认拒绝。
- `is_allowed=false` 默认拒绝。
- `need_approval=true` 当前阶段默认拒绝。
- `estimatedRows > maxRows` 默认拒绝。

场景：

- 默认 `QUERY`。
- 详情接口使用 `DETAIL`。
- 导出接口使用 `EXPORT`。
- SQL 过滤当前阶段所有场景都生效。
