## 2. 背景与目标

### 2.1 背景

当前业务服务已经具备日志检索、告警、资产管理等能力，但数据访问控制仍分散在各服务中，存在以下问题：

- 权限逻辑容易散落在业务代码中，后续维护成本高
- 不同服务的权限口径不一致，容易产生越权风险
- 缺少统一的字段脱敏、导出控制和审计能力
- 高并发场景下如果每次请求都远程调用权限中心，性能不可接受
- 后续存在 Rust 化演进诉求，需要设计上提前解耦

### 2.2 建设目标

本方案希望实现以下目标：

- 在不大改现有业务服务代码的前提下，统一接入数据权限能力
- 支持 PostgreSQL 与 Doris 两类查询场景的权限过滤
- 支持租户、组织、角色、用户、资产组、数据等级等维度的控制
- 支持字段级权限、动态脱敏、导出控制与访问审计
- 支持高并发下的低延迟权限获取与快速失效
- 为后续 Rust SDK 或多语言 SDK 演进预留标准化接口

## 3. 设计原则

### 3.1 核心原则

1. `默认拒绝`：无权限、权限为空、上下文获取失败时，一律拒绝或返回空结果。
2. `统一收口`：权限策略由权限服务统一管理，业务服务只负责接入执行。
3. `低侵入接入`：通过 SDK、拦截器、AOP，尽量不改业务代码。
4. `高性能优先`：通过本地缓存 + 事件失效降低远程调用开销。
5. `兼容演进`：先满足 Java MVP，再抽象出可复用模型支持 Rust 版。
6. `服务端强校验`：权限控制必须在服务端执行，前端仅做交互辅助。
7. `统一上下文传递机制`：SDK 采用‘Web 拦截器先行提取、线程绑定（ThreadLocal）全局传递’的设计模式。后续的 SQL 重写插件、Jackson 脱敏处理器、AOP 导出拦截器以及审计组件，均直接从当前线程上下文（Context Holder）中获取当前操作的用户身份，确保整个拦截链路的身份口径绝对一致

### 3.2 关键设计结论

- 采用`权限服务 + SDK + 业务服务`三层架构
- 权限服务负责`策略管理、上下文计算、缓存维护、审计收口`
- SDK 负责`上下文获取、SQL 重写、字段脱敏、导出拦截、审计上报`
- 业务服务保持`业务逻辑纯净`，不硬编码权限判断细节

## 4. 总体方案概述

### 4.1 方案定位

本方案不是把每次权限判断都交给中心服务实时完成，而是由权限服务提前计算用户可访问的权限上下文，业务侧 SDK 在本地完成高频执行。

这意味着：

- 管理集中在权限服务
- 执行面下沉到业务服务本地
- 变更通过缓存失效和事件广播同步

### 4.2 三层架构说明

#### 1. 业务服务

业务服务是数据的真正宿主，例如：

- [ ] 日志检索服务

* 告警中心
* 事件中心
* 资产管理
* 报表中心

业务服务本身不关心用户有哪些资产组权限，也不自己拼装 SQL 权限条件，只负责原始业务查询、写入和展示。

#### 2. Data Permission SDK

SDK 以内嵌方式接入业务服务，是权限执行的统一入口，负责：

- 获取并缓存当前用户权限上下文
- 在 SQL 执行前自动追加权限过滤条件
- 在接口返回前执行字段脱敏
- 在导出前进行权限检查和条数限制
- 异步记录审计日志

#### 3. 权限服务

权限服务是集中式管理与计算中心，负责：

- 策略配置管理
- 权限上下文计算与打平
- Kafka 失效广播
- 审计日志消费与落库

## 5. 技术架构选型

### 5.1 组件选型

| 组件       | 选型                  | 用途               | 选型原因               |
| -------- | ------------------- | ---------------- | ------------------ |
| 权限服务     | Java Spring Boot    | 权限管理、上下文计算、接口服务  | 团队开发效率高，适合快速完成 MVP |
| Java SDK | Spring Boot Starter | 业务服务低侵入接入        | 与现有 Java 业务服务集成成本低 |
| Rust SDK | 后续演进                | Rust 业务服务权限接入    | 复用统一接口与模型，适配生产版    |
| 数据库      | PostgreSQL          | 策略、绑定关系、上下文快照、审计 | 事务能力强，适合结构化元数据     |
| 本地缓存     | Caffeine            | L1 上下文缓存         | JVM 内访问极快，适合高频查询   |
| 消息队列     | Kafka               | 缓存失效广播、审计日志异步收口  | 吞吐高，适合广播与削峰        |
| SQL 解析   | JSqlParser          | SQL AST 改写       | 便于安全地追加过滤条件        |
| 序列化脱敏    | Jackson 扩展          | 接口返回字段动态脱敏       | 可统一处理 DTO 输出       |
| 导出拦截     | Spring AOP          | 导出前权限预检          | 实现简单，接入侵入小         |

### 5.2 为什么不把权限校验完全放在权限服务

如果每一次列表查询、详情查看、日志搜索都同步调用权限服务：

- 权限服务会成为高并发瓶颈
- 日志检索类请求量大，网络开销会被放大
- 查询链路增加一次远程调用，P95延迟(长尾延迟)明显上升
- 某个中心组件抖动会影响全部业务服务

因此，本方案采用`中心管理、边缘执行`模式：权限服务负责计算和管理，SDK 在业务服务本地执行高频校验。

### 5.3 为什么采用二级缓存

选择`Caffeine(L1) 的原因如下：

- L1 命中时接近本地内存访问，适合日志检索、列表分页等高频场景
- 配合 Kafka 广播实现主动失效，平衡性能与一致性

### 5.4 什么是 SQL AST？为什么要用它？

SQL AST（Abstract Syntax Tree，抽象语法树），简单来说就是“把一段文本格式的 SQL 语句，翻译成计算机在内存中看得懂的‘树状结构对象’。”

为什么要用 AST（相比传统字符串拼接的优势）？
在过去，我们要给一条查询语句追加权限条件，往往采用直接拼接字符串的做法。但在企业级数据权限场景下，这种做法有两大致命缺陷：
a. 安全隐患极高：直接拼接极易导致 SQL 注入漏洞。
b. 语法容易崩溃：业务发出的原始 SQL 往往非常复杂，可能带有 UNION、GROUP BY、ORDER BY、LIMIT 或多层子查询。如果盲目在末尾拼上 AND tenant\_id = 'xxx'，会导致 SQL 直接报语法错误，导致线上业务不可用。

用 AST 的核心优势
a. 手术刀式的精准操控：SDK 拦截到业务 SQL 后，通过解析器（如 Java 的 JSqlParser 或 Rust 的 sqlparser-rs）将其转化为一颗“树”。我们可以极其精准地找到这棵树上的“表节点”和“条件节点”，并在合适的位置强行挂载我们需要的权限过滤参数（如 AND tenant\_id = ?）。
b. 原生防注入与零语法错误：因为修改的是对象树，最后由引擎重新还原出来的 SQL 结构 100% 保证合法，且全部采用参数化预编译（? 占位符），从根本上杜绝了 SQL 注入。

如何实现对 PostgreSQL 和 Doris 的自动过滤？
🔄 核心落地流程
我们采用的是“在底层拦截，根据数据源动态适配，在业务本地重写”的架构方案。
a. 统一拦截：当业务服务发出查询请求时，SDK 内置的 MyBatis 或 JDBC 插件会第一时间在底层把 SQL 拦截下来。
环境感知：SDK 会读取当前的配置，判断当前查询的目标物理表到底属于哪个计算引擎（例如：识别到告警表，对应 PostgreSQL；识别到海量日志表，对应 Doris）。
b. 策略注入：SDK 从本地的二级缓存中提取当前用户的权限上下文（包含所属租户、被授权的资产组等）。

通过 AST 语法树技术，在不需要业务开发写任何一行权限代码的前提下，由 SDK 在底层透明地把‘权限网’织进 SQL 里。它既兼顾了 PostgreSQL 的方言特性，又通过字段冗余设计保障了 Doris 在海量日志检索下的高吞吐与低延迟。这套机制是目前微服务架构中性能损耗最少、对业务侵入最低的解法.

## 7. 模块划分

### 7.1 权限服务模块

### 7.1.1 策略配置管理模块

职责：

- 管理数据权限策略
- 管理授权主体与资源绑定关系
- 管理字段脱敏规则
- 管理导出规则
- 提供管理端配置 API

### 7.1.2 上下文计算引擎

职责：

- 聚合用户直接授权、角色授权、部门授权、用户组授权
- 解析组织树、资产组等
- 按租户维度生成最终权限上下文
- 对空权限场景打上`deny_all`标记

核心规则：

- `deny` 优先于 `allow`
- 用户级授权优先级高于角色级授权
- 数据范围合并后进行去重和打平
- 上下文生成失败时返回默认拒绝

### 7.1.3 上下文分发与缓存模块

职责：

- 对 SDK 提供上下文查询与权限校验接口
- 维护 version 版本号
- 在策略变更时广播失效事件

### 7.1.4 统一审计模块

职责：

- 接收 SDK 异步上报的审计日志
- 批量落盘至 PostgreSQL / Doris审计表
- 为安全合规提供追溯能力

***

### 7.2 SDK 模块

### 7.2.1 Context Holder

职责：

- 保存当前请求的用户权限上下文
- 向后续 SQL 重写、脱敏、导出校验提供统一上下文入口

建议实现：

- Java 使用 `ThreadLocal`
- Rust 版本可抽象为 request scoped context

### 7.2.2 Context Manager

职责：

- 负责 L1/远程回源二层获取逻辑
- 维护上下文过期时间和版本
- 接收 Kafka 失效消息并清理 L1缓存

### 7.2.3 SQL Rewrite Engine

职责：

- 识别 PostgreSQL / Doris 查询类型
- 基于 AST 向 SQL 中安全追加权限条件
- 统一处理`tenant_id`、`dept_id`、`asset_group_id`、`created_by`等过滤字段

设计要求：

- 必须使用参数化条件，不允许字符串拼接
- 权限为空时改写为 `WHERE 1 = 0`
- 不允许业务 SQL 覆盖平台权限过滤

### 7.2.4 Data Masking Engine

职责：

- 在 DTO/VO 序列化阶段识别敏感字段
- 根据当前上下文和字段策略进行 show/mask/hide/deny 处理

### 7.2.5 Export Guard

职责：

- 检查是否允许导出
- 检查预计导出条数是否超限
- 判断是否需要审批
- 导出时强制脱敏
- 记录导出审计

### 7.2.6 Audit Reporter

职责：

- 对查询、拒绝、导出、配置变更等关键行为生成审计事件
- 异步投递至 Kafka
- 避免审计逻辑阻塞主链路

## 8. 关键业务流程

### 8.1 权限上下文获取流程

1. 请求进入业务服务，SDK 从请求中提取身份信息
2. 优先查询本地 Caffeine 缓存
3. 未命中则调用权限服务计算上下文
4. 权限服务合并策略并生成最终上下文
5. 上下文写入本地缓存并返回业务线程

### 8.2 查询权限控制流程

1. 业务服务生成原始 SQL
2. MyBatis/JDBC 插件拦截 SQL
3. SDK 根据资源类型找到对应过滤字段映射
4. 将权限条件通过 AST 追加至原始 SQL
5. 权限为空时重写为无结果 SQL
6. 执行查询并记录审计摘要

### 8.3 字段脱敏流程

1. 业务接口返回 DTO
2. SDK 获取当前场景，例如 `QUERY`、`DETAIL`、`EXPORT`
3. 读取字段策略
4. 对敏感字段执行 show、mask、hide、deny
5. 返回处理后的响应

### 8.4 导出控制流程

1. 业务接口触发导出
2. SDK 校验导出开关
3. 先执行 `COUNT` 或预估条数检查
4. 超出阈值时拒绝或进入审批
5. 导出过程中强制应用字段脱敏
6. 记录导出审计和最终结果

## 9. 权限模型设计

### 9.1 授权主体

支持以下主体：

- 用户
- 角色
- 部门
- 用户组
- 租户级管理员

### 9.2 权限资源

支持以下资源维度：

- 租户
- 部门
- 资产组
- 数据等级
- 资源类型
- 字段
- 导出动作

### 9.3 权限优先级

建议采用如下优先级规则：

1. `DENY` 优先于 `ALLOW`
2. 用户级策略优先于角色级策略
3. 角色级策略优先于部门级策略
4. 同级策略按 `priority` 数值排序
5. 策略合并后统一生成最终上下文

## 10. 数据库设计

### 10.1 核心表概览

| 表名                        | 说明       |
| ------------------------- | -------- |
| `data_permission_policy`  | 数据权限主策略表 |
| `sys_asset_group`         | 资产组信息表   |
| `sys_user_group`          | 用户组信息表   |
| `data_field_policy`       | 字段策略表    |
| `data_policy_export_rule` | 导出控制策略表  |
| `data_permission_audit`   | 审计日志表    |

说明：用户、角色、部门、组织树、租户、资产组等主数据建议复用平台现有基础表，不在权限服务中重复建设。

### 10.2 数据权限主策略表

本表为数据权限系统的核心控制面表。用于存储安全策略的定义、授权主体（用户、角色、部门、用户组）与数据范围、资源维度的多维交叉绑定关系

```sql
CREATE TABLE data_permission_policy (
    id BIGSERIAL PRIMARY KEY,
    policy_name VARCHAR(128) NOT NULL,
    description VARCHAR(256),
    
    -- 【授权主体立体交叉组合】
    tenant_id VARCHAR(64) NOT NULL,          -- 所属租户边界
    target_type  VARCHAR(64) NOT NULL,      -- 授权目标类型（USER、ROLE、DEPT）
    target_dept_ids JSONB,            -- 目标部门（可为空） ["dept003", "dept004"] 
    target_role_ids JSONB,             -- 目标角色（可为空） ["role003", "role004"] 
    target_user_ids JSONB,             -- 目标用户（可为空） ["user003", "user004"] 
    target_user_group_ids JSONB,       -- 目标用户组（可为空） ["user_group003", "user_group004"] 

    -- 【组织维度数据范围控制】
    dept_scope_type VARCHAR(32) NOT NULL,   -- ALL(全部), CURRENT(本部门), SUB(本部门及下级), SELF(仅本人), CUSTOM(指定部门)
    custom_dept_ids JSONB,                  -- 当dept_scope_type为CUSTOM时，存储指定的部门ID数组 ["d003", "d004"]

    -- 【数据等级控制】
    data_level_scope JSONB NOT NULL,       -- 可访问的数据等级，例如： ["public","internal","sensitive"]
    
    -- 【资产组维度】：允许访问的资产组集合
    -- 存储如：["ag_001", "ag_002"] 或 ["*"]
    asset_group_scope JSONB,

    -- 资源对象（限制访问对象）
    resource_object JSONB NOT NULL,             -- 资源对象。示例: {"doris":["security_log"],"postgre_sql":["asset"]}  或  {"doris":["security_log"],"postgre_sql":["asset"]}
    
    -- 【字段级策略】：针对不同资源类型的字段脱敏/隐藏规则
    -- 存储如：["field_policy_id1","field_policy_id2"]
    field_policy_ids JSONB NOT NULL,
    
    -- 【导出控制策略】：针对不同资源类型的导出限制
    -- 存储如：["export_policy_id1","export_policy_id2"]
    export_policy_ids JSONB NOT NULL,
    
    -- 【控制开关】
    effect VARCHAR(16) DEFAULT 'allow',     -- allow / deny (deny优先)
    priority INT DEFAULT 100,               -- 优先级，数字越大越优先
    is_enabled BOOLEAN DEFAULT false,        -- 是否生效
    created_at TIMESTAMP DEFAULT now(),  -- 创建时间：授权发生的时间
    created_by VARCHAR(64) NOT NULL,    -- 创建人：发起授权的用户 ID
    updated_at TIMESTAMP,  -- 更新时间：授权发生的时间
    updated_by VARCHAR(64)     -- 更新人：发起授权的用户 ID
);
```

### 10.5 字段策略表

本表用于定义细粒度的列级别（Field-level）数据控制规则。当用户请求敏感资源（如日志、资产详情）时，SDK 根据本表配置的脱敏、隐藏、禁止规则对特定字段进行动态动态处理。

    id BIGSERIAL PRIMARY KEY,
    tenant_id VARCHAR(64) NOT NULL,    -- 租户 ID：所属租户边界

### 10.7 审计日志表

本表为数据权限系统的数据合规与行为追踪表。采用只读追加写（Append-Only）设计，真实记录每次触发权限拦截、SQL 改写、敏感字段脱敏及导出动作时的完整上下文，提供完整的防抵赖审计铁证。

```sql
CREATE TABLE data_permission_audit (
    id BIGSERIAL PRIMARY KEY,
    tenant_id VARCHAR(64) NOT NULL,                              -- 租户ID
    user_id VARCHAR(64) NOT NULL,                                -- 操作员用户ID
    action VARCHAR(32) NOT NULL,                                 -- 审计动作。枚举值: QUERY(列表), DETAIL(详情), EXPORT(导出), DENIED(阻断), POLICY_CHANGE(策略变更)
    resource_type VARCHAR(64) NOT NULL,                         -- 资源类型。示例: DORIS / POSTGRESQL
    resource_id VARCHAR(64) NOT NULL,                           -- 资源对象/表名。示例: security_log / alert / event / report
    filter_sql TEXT,                                             -- SDK 最终重写注入的参数化 SQL 判定条件。示例: "tenant_id = ? AND asset_group_id IN (?)"
    estimated_rows INT DEFAULT 0,                                -- 预估/受影响数据条数（COUNT 预检结果）
    rows_affected INT DEFAULT 0,                                 -- 实际导出或返回条数
    result VARCHAR(16) NOT NULL,                                 -- 审计结果。枚举值: ALLOWED(放行), DENIED(默认拒绝/阻断), SUCCESS(变更成功)
    client_ip VARCHAR(64),                                       -- 客户端真实 IP 地址
    created_at TIMESTAMP WITH TIME ZONE DEFAULT now() NOT NULL   -- 审计生成时间（不允许修改）
);

-- 审计表只做高效追加，建立时间复合索引应对安全审计报表查询
CREATE INDEX idx_dpa_query ON data_permission_audit (created_at DESC, tenant_id, user_id);
```

说明：

- 不可篡改。必须记录谁、在什么时间、执行了什么查询、注入了什么过滤条件。

- `QUERY`
- `DETAIL`
- `EXPORT`

## 11. 接口设计

### 11.1 接口补齐结论

当前功能不仅需要“权限管理”和“权限校验”，还需要支撑 SDK 在本地执行权限过滤、字段脱敏、导出拦截和审计上报。因此接口需要分为两类：

| 分类       | 用途                                | 调用方                    |
| -------- | --------------------------------- | ---------------------- |
| SDK 执行接口 | 获取上下文、检查资源访问权限                    | Java SDK、Rust SDK、业务服务 |
| 管理端接口    | 管理策略、资产组、用户组、字段策略、导出策略、审计日志、上下文刷新 | 权限管理后台、系统管理服务          |

完整接口契约已整理为独立文档：`最终版/接口设计文档V2.md`。本章保留设计原则、接口总览和核心示例，开发实现以独立接口文档为准。

### 11.2 通用规范

### 11.6 错误码

| code                         | HTTP 状态码 | 说明        | SDK 建议处理                |
| ---------------------------- | -------- | --------- | ----------------------- |
| `0`                          | 200      | 成功。       | 正常处理。                   |
| `DP_400_PARAM_INVALID`       | 400      | 请求参数错误。   | 拒绝本次访问并记录审计。            |
| `DP_401_UNAUTHORIZED`        | 401      | 未认证或令牌失效。 | 拒绝本次访问。                 |
| `DP_403_FORBIDDEN`           | 403      | 调用方无接口权限。 | 拒绝本次访问。                 |
| `DP_404_NOT_FOUND`           | 404      | 资源不存在。    | 管理端提示不存在；SDK 按无权限处理。    |
| `DP_409_VERSION_CONFLICT`    | 409      | 上下文版本冲突。  | SDK 重新拉取上下文。            |
| `DP_422_POLICY_INVALID`      | 422      | 策略配置不合法。  | 管理端提示修正配置。              |
| `DP_429_TOO_MANY_REQUESTS`   | 429      | 调用过于频繁。   | SDK 使用本地缓存；无缓存时默认拒绝。    |
| `DP_500_INTERNAL_ERROR`      | 500      | 服务内部错误。   | 默认拒绝或返回空结果。             |
| `DP_503_CONTEXT_UNAVAILABLE` | 503      | 权限上下文不可用。 | 默认拒绝或将 SQL 改写为 `1 = 0`。 |

错误响应示例：

```json
{
  "code": "DP_503_CONTEXT_UNAVAILABLE",
  "message": "permission context is unavailable",
  "trace_id": "trc-20260629-9999",
  "data": {
    "deny_all": true,
    "reason": "context_compute_failed"
  }
}
```

## 12. 业务服务接入设计

### 12.1 Java 服务接入方式

建议以 `Spring Boot Starter` 形式提供能力：

- 自动装配 Web 拦截器
- 自动装配 MyBatis/JDBC 插件
- 自动装配 Jackson 脱敏模块
- 自动装配 Export Guard AOP
- 自动装配 Kafka 消费与审计上报能力

配置示例：

```yaml
data-permission:
  enabled: true
  service-url: http://data-permission-service
  cache:
    l1-expire-seconds: 60

  kafka:
    invalidate-topic: auth-context-invalidate
    audit-topic: data-permission-audit-topic
  intercept:
    resources:
      security_log:
        engine: doris
        table-names: [normalized_security_log]
      alert:
        engine: postgresql
        table-names: [t_alert_event]
```

### 12.2 Rust 服务接入方式

Rust 生产版建议不复刻 Java 框架实现，而是复用统一协议：

- 复用权限服务 HTTP/gRPC 接口
- 复用 Kafka 事件协议
- 复用上下文 JSON 模型
- 使用 `sqlparser-rs` 替代 `JSqlParser`
- 使用中间件方式植入请求上下文

这样可以做到：

- Java MVP 先落地
- Rust 版实现只替换执行器，不推翻整体架构

## 13. 安全设计

### 13.1 默认拒绝

以下场景必须默认拒绝：

- 无法识别用户身份
- 上下文获取失败
- 上下文为空
- 策略未命中且无显式放行
- 导出预检失败

### 13.2 SQL 安全重写

必须满足以下要求：

- 使用 AST 改写，不直接字符串拼接
- 生成参数化条件，如 `tenant_id = ?`
- 不允许业务方自带的 tenant 参数覆盖服务端上下文
- 资源无权限时返回空结果，不执行原始全量查询

### 13.3 字段安全

- 敏感字段默认不明文返回
- 导出场景可配置为强制脱敏
- 详情与列表可以使用不同字段策略

### 13.4 审计与防抵赖

- 查询、导出、拒绝、变更均需留痕
- 审计日志采用异步投递，避免主链路阻塞
- 审计表只追加写，不向普通用户开放更新删除能力

***

## 14. 性能与可用性设计

### 14.1 性能目标

建议目标如下：

- 上下文获取接口 P95 小于 100ms
- 本地缓存命中时上下文获取耗时接近 0ms
- 权限校验接口 P95 小于 50ms
- 高频日志查询场景不因权限接入出现明显性能下降

### 14.2 性能关键点

- L1 Caffeine 缓存降低 JVM 内重复查询成本
- 优先使用 `asset_group_id`等低基数字段过滤
- 避免生成超长 `asset_id IN (...)` 条件
- 管理员 `ALL` 权限不生成无意义冗余条件

### 14.3 高可用设计

- 权限服务保持无状态，可水平扩展
- 权限服务与 Kafka 采用解耦设计，失效广播失败不影响主数据提交
- SDK 审计发送采用异步容错，避免主业务线程阻塞

***

## 15. 设计优势

### 15.1 对业务服务友好

- 接入方式低侵入，不要求大规模改造现有查询代码
- 权限逻辑从业务代码中剥离，降低长期维护成本

### 15.2 对安全性友好

- 默认拒绝、防注入、字段脱敏、导出控制、审计追踪形成闭环
- 服务端统一收口，避免前端绕过

### 15.3 对性能友好

- 高并发请求主要在业务服务本地完成权限执行
- 权限服务不成为每次查询的同步瓶颈

### 15.4 对演进友好

- Java SDK 与 Rust SDK 共享统一协议与模型
- 后续新增资源类型、动作类型、脱敏规则时可平滑扩展

***

## 16. 风险与边界

### 16.1 Doris 表模型要求

为了保证查询性能，Doris 日志、告警、事件表建议冗余以下权限过滤字段：

```text
tenant_id, asset_group_id, dept_id, data_level
```

否则后续权限过滤会依赖 Join 或子查询，导致性能不可控。

### 16.2 业务表字段映射需要治理

不同业务表对“部门、资产组、创建人”等字段命名可能不一致，需要在 SDK 中建立资源字段映射配置，否则很难统一重写逻辑。

### 16.3 一期建议聚焦 MVP

一期优先建设以下能力：

- 租户隔离
- 部门范围
- 资产组权限
- SQL 自动过滤
- 字段脱敏
- 导出控制
- 审计留痕

***

## 17. 分阶段落地建议

### 17.1 第一阶段：Java MVP

目标：

- 权限服务使用 Java Spring Boot
- Java 业务服务通过 Starter 接入
- 完成 PostgreSQL / Doris SQL 过滤
- 完成字段脱敏、导出控制、缓存失效、审计

### 17.2 第二阶段：平台化增强

目标：

- 完善管理后台页面
- 增加策略模拟验证
- 增加审批流对接
- 增加更多资源类型和策略模板

### 17.3 第三阶段：Rust 生产版演进

目标：

- Rust 服务复用统一权限协议
- 抽象多语言 SDK 接口规范
- 保持权限中心和事件协议不变，仅替换本地执行实现

***

## 18. 结论

本方案推荐采用`权限服务集中管理 + SDK 分布式执行 + 二级缓存 + Kafka 失效广播 + 审计异步收口`的总体架构，用统一的数据权限模型为现有业务服务补齐权限管理与权限检验能力。

该方案的核心价值在于：

- 对现有业务系统改造成本可控
- 能同时覆盖 PostgreSQL 与 Doris 场景
- 具备较好的安全性、性能与扩展性平衡
- 可平滑支撑 Java MVP 到 Rust 生产版演进

如果按该方案落地，建议优先从`权限上下文模型`、`SQL 过滤字段映射`、`Java SDK 接入规范`、`策略管理接口`四部分启动实施，这四项决定了一期是否能真正落地。
