### 设计

拆分分三个部分：
1. 业务服务 (Business Service)
   角色：数据的最终运行方与宿主（如日志系统、告警微服务、资产管理系统）。

核心特性：本身不包含任何权限校验的硬编码，不知道当前用户能看哪些资产，也不关心怎么拼接 WHERE 条件，只专注于执行业务本身的增删改查。

对其他二者的依赖：强依赖 SDK 内嵌于自己的进程中。

2. Data-Perm SDK (核心纽带)
   角色：权限控制的前线执行官（以 Spring Boot Starter Crate 的形式直接内嵌于业务服务内部）。

核心特性：它是业务服务与权限服务之间的减震器与执行桥梁。

向上（对权限服务）：通过网络与权限服务打交道。它负责向权限服务发起 RPC 请求换取用户权限上下文，并潜伏在业务服务里监听权限服务的 Kafka 失效广播，随时精准清空自己内存里的旧权限。

向下（对业务服务）：在业务服务内部发挥“底层隐形插件”的作用。在业务服务即将发往 PostgreSQL/Doris 执行 SQL 时，利用 JSqlParser 强行把从缓存拿到的权限条件追加进去；在业务服务返回数据给前端时，自动执行脱敏；在触发导出时自动计数阻断，并用统一格式往日志中打印一行审计 JSON。

3. 权限服务 (Auth Service)
   角色：全局策略的大脑与后台管理端。

核心特性：纯无状态、不参与业务运行。当全站高并发检索 Doris 日志时，权限服务自己是坐在后台“喝茶”的，请求根本不会到达权限服务，从而保证了中台不会成为高并发下的性能单点瓶颈。

对其他二者的依赖：它只负责在管理员修改配置时，往 Redis 写入最新快照，并向 Kafka 发送一条“XX 用户权限变了，大家清缓存”的广播。同时，它扮演一个安静的“数据收口者”，通过消费者把 SDK 发送出来的审计消息批量捡起来，写入审计库备查。

##### 3. 注意点


异常分支的闭环（默认拒绝）：在微观流程图中，一旦 Core 服务宕机或 Redis 挂掉导致 SDK 拿不到上下文，SDK 不会崩溃，也不会直接放行原 SQL，而是原地将 SQL 篡改为 WHERE 1 = 0。这样既保护了下游 Doris 不会被意外越权脱裤，又保证了系统的高可用性（宁可报错，绝不漏报）。

AST 的参数化安全重写：在 AST 节点拼接时，图中的“权限条件节点”在底层转化为 SQL 时，必须是 tenant_id = ? 这种带占位符的预编译形式，随后由 SDK 将当前上下文的值填入 Parameter 列表中。绝不能直接拼接字符串（如 tenant_id = ' + id + '），在流程中切断了任何 SQL 注入的可能。



### 数据权限管理服务（微服务）

负责权限策略的管理、计算、下发和审计。它主要包含以下核心模块和功能：

① 策略配置管理模块 (Admin Module)
多租户隔离配置：定义租户边界，管理租户的初始化权限范围。

交叉矩阵授权功能：提供统一的策略配置 API，将安全主体（用户/角色/部门）与数据资源维度（资产组范围、日志源 ID、物理组织架构 scope）进行多维矩阵交叉绑定。

字段脱敏与级联控制配置：维护哪些字段（如手机号、Token）在什么角色下需要执行何种策略（隐藏/正则打码/明文），配置各资源的导出开关及最大行数阈值（如普通用户 5 万条，管理员 50 万条）。

② 策略打平与关系计算引擎 (Flattening Engine)
组织架构拓扑打平：当配置“本部门及下级部门”时，引擎负责通过树遍历算法，将继承关系动态计算为平铺的 List<dept_id>。

资产/日志源矩阵打平：将角色所关联的资产组继承关系动态解析，消除复杂的数据库 JOIN 关系，生成扁平化的标签列表。

③ 上下文分发与缓存同步模块 (Context & Cache Module)
无状态上下文构建：提供高性能的 gRPC / HTTP 接口，供各个微服务或 SDK 首次接入时换取打平后的权限 JSON 报文。

分布式缓存维护：负责将计算好的上下文同步写入 Redis，并采用乐观版本号（Version）控制。

失效事件广播机制：当管理员修改权限、组织架构变更时，该模块负责向环境中的 Kafka 发送一条轻量级的失效广播消息（如 {"user_id": "u001", "version": 14}），通知全站物理节点清除本地缓存。

④ 统一审计接收与存储模块 (Audit Ingestion Module)
Kafka 审计流消费者：常驻后台的高吞吐消费者，专门订阅统一的 data-permission-audit-topic。

批量落盘引擎：采用 Insert Only 追加写入模式，将多语言 SDK 异步上报的审计 JSON 批量刷入 PostgreSQL 分区表或 ClickHouse/Doris。

###  表设计
##### 1.1 权限上下文表 (user_data_permission_context)

设计核心：用于缓存用户最终计算出的权限边界，避免每次业务查询都去扫描复杂的策略表。使用 PostgreSQL 的 JSONB 类型方便存储数组和动态策略。  
字段名类型说明

```
CREATE TABLE user_data_permission_context (
    id BIGSERIAL PRIMARY KEY,
    
    -- 核心检索三元组（多租户、多部门支持）
    user_id VARCHAR(64) NOT NULL,          -- 用户ID
    tenant_id VARCHAR(64) NOT NULL,        -- 当前激活的租户ID
    
    -- 组织维度：允许访问的部门集合
    -- 组织范围最终计算出的部门ID集合（比如把“本部门及下级”打平后的ID列表）
    dept_scope JSONB NOT NULL,              -- 格式: ["d001", "d002", "d003"] 或 ["*"]
    
    -- 资产组维度：允许访问的资产组集合
    -- 存储如：["ag_001", "ag_002"] 或 ["*"]
    asset_group_scope JSONB,
    
    -- 日志源维度：允许访问的日志源集合
    -- 存储如：["ls_waf", "ls_firewall"] 或 ["*"]
    log_source_scope JSONB,
    
    -- 数据等级维度：允许访问的数据密级
    -- 存储如：public/internal/sensitive
    data_level_scope  VARCHAR(32) NOT NULL,
    
    -- 字段级策略：针对不同资源类型的字段脱敏/隐藏规则
    -- 存储如：[{"table":"security_log","rule":{[{"scene":"LIST","fields":["key": "phone", "permission_type": "mask", "custom_regex": "xxx"}]}]}]
    field_policy_json JSONB NOT NULL,
    
    -- 导出控制策略：针对不同资源类型的导出限制
    -- 存储如：[{"table":"security_log","rule":{"allowed": true, "max_rows": 5000, "need_approval": true}}]
    export_policy_json JSONB NOT NULL,
    
    -- 缓存控制与版本
    version BIGINT DEFAULT 1 NOT NULL,     -- 版本号，用于缓存一致性校验
    expired_at TIMESTAMP NOT NULL,          -- 缓存过期时间
    created_at TIMESTAMP DEFAULT now(),
    updated_at TIMESTAMP DEFAULT now()
    
);

-- 核心联合唯一索引：确保一个用户在【特定租户+特定部门】下只有一条最终生效的上下文
CREATE UNIQUE INDEX uk_user_tenant_dept ON user_data_permission_context (user_id, tenant_id, dept_id);

-- 检索索引优化性能
CREATE INDEX idx_ctx_expired ON user_data_permission_context (expired_at);

```

##### 1.2. 数据权限主策略表

设计核心：记录主体（用户/角色/部门）对具体资源（资产组/日志源..）的直接授权关系。

```
CREATE TABLE data_permission_policy (
    id BIGSERIAL PRIMARY KEY,
    policy_name VARCHAR(128) NOT NULL,
    description VARCHAR(256),
    
    -- 【授权主体立体交叉组合】
    tenant_id VARCHAR(64) NOT NULL,          -- 所属租户边界（或指定ALL表示平台级）
    target_dept_id VARCHAR(64),             -- 目标部门（可为空，为空代表不限部门）
    target_role_id VARCHAR(64),             -- 目标角色（可为空）
    target_user_id VARCHAR(64),             -- 目标用户（可为空）
    target_user_group_id VARCHAR(64),       -- 目标用户组（可为空）

    -- 【组织维度数据范围控制】
    dept_scope_type VARCHAR(32) NOT NULL,   -- ALL(全部), CURRENT(本部门), SUB(本部门及下级), SELF(仅本人), CUSTOM(指定部门)
    custom_dept_ids JSONB,                  -- 当dept_scope_type为CUSTOM时，存储指定的部门ID数组 ["d003", "d004"]

    -- 【数据等级控制】
    data_level_scope VARCHAR(32) NOT NULL,       -- 可访问的数据等级数组，例如：public/internal/sensitive
    
    -- 【资产组维度】：允许访问的资产组集合
    -- 存储如：["ag_001", "ag_002"] 或 ["*"]
    asset_group_scope JSONB,
    
    -- 【日志源维度】：允许访问的日志源集合
    -- 存储如：["ls_waf", "ls_firewall"] 或 ["*"]
    log_source_scope JSONB,
    
    -- 【字段级策略】：针对不同资源类型的字段脱敏/隐藏规则
    -- 存储如：["field_1","field_2"]
    field_policy_json JSONB NOT NULL,
    
    -- 【导出控制策略】：针对不同资源类型的导出限制
    -- 存储如：["export_id1","export_id2"]
    export_policy_json JSONB NOT NULL,
    
    -- 【控制开关】
    effect VARCHAR(16) DEFAULT 'allow',     -- allow / deny (deny优先)
    priority INT DEFAULT 100,               -- 优先级，数字越大越优先
    is_enabled BOOLEAN DEFAULT true,        -- 是否生效
    created_at TIMESTAMP DEFAULT now()  -- 创建时间：授权发生的时间
    created_by VARCHAR(64) NOT NULL,    -- 创建人：发起授权的用户 ID
    updated_at TIMESTAMP  -- 更新时间：授权发生的时间
    updated_by VARCHAR(64)     -- 更新人：发起授权的用户 ID
);
```

##### 1.7 审计日志表 (data_permission_audit)

设计核心：不可篡改。必须记录谁、在什么时间、执行了什么查询、注入了什么过滤条件。

```
SQLCREATE TABLE data_permission_audit (
    id BIGSERIAL PRIMARY KEY,
    tenant_id VARCHAR(64) NOT NULL,     -- 租户 ID：操作发生时用户所属的租户
    user_id VARCHAR(64) NOT NULL,        -- 操作用户 ID：发起请求或变更的真实操作员身份
    action VARCHAR(32) NOT NULL,          -- 操作行为类型：QUERY(查询), EXPORT(导出), POLICY_CHANGE(策略变更)
    resource_type VARCHAR(64) NOT NULL,   -- 涉及资源对象：如 t_security_log 或 POLICY_ENGINE
    filter_sql TEXT,                     -- 实际注入的 SQL / 阻断逻辑：记录 SDK 最终重写注入的参数化条件或拦截拦截详情
    estimated_rows  INT,                 -- 预估/受影响数据条数：导出时，记录 COUNT(1) 预检出的条数，用于追溯超限拦截
    result VARCHAR(16) NOT NULL,         -- 审计结果状态：ALLOWED(放行), DENIED(阻断/默认拒绝), SUCCESS(变更成功)
    rows_affected INT DEFAULT 0,         -- 影响/导出条数
    client_ip VARCHAR(64),               -- 客户端真实 IP：发起请求的终端网络地址，防止越权伪造
    created_at TIMESTAMP DEFAULT now()   -- 审计记录时间：事件发生的服务器标准时间（不可篡改）
);

```

##### 1. SDK 的定位与四大核心功能

    SDK 是嵌套在各个业务微服务（如告警中心、日志检索）内部的轻量级安全组件。它不负责策略的制定，只负责策略的刚性执行与落地。它包含以下四大核心功能：


SDK 的核心设计与功能
Data Permission SDK 是一个 Spring Boot Starter。它的核心职责是：对业务代码零侵入/低侵入，在底层帮业务把事情办了。

它主要包含四大核心模块：

Context Holder：基于 ThreadLocal，存储当前请求的用户权限上下文（TenantId, DeptScope, AssetGroupScope 等）。

SQL Rewrite Engine（AST 重写引擎）：基于 JSqlParser。拦截业务层即将发往 PostgreSQL 或 Doris 的 SQL，解析成语法树，强行在 WHERE 后面追加权限过滤条件。

Data Masking Engine（脱敏引擎）：基于 Jackson 序列化拦截器。当控制层返回 DTO 给前端时，自动扫描带有 @DataMask 注解的字段，结合当前用户上下文执行脱敏。

Export & Audit Guard（导出与审计卫兵）：提供 AOP 切面，在执行导出方法前，预检行数是否超标，并异步发送审计日志。

### 考虑点

#### 1 SDK为什么引入二级缓存架构 
在高并发的日志检索或告警涌入场景下，网络 IO 和 Redis 带宽会被瞬间打爆。为了把延迟控制在一定的范围以内，我们必须把权限上下文直接放进业务系统的 JVM 内存中。

1. 二级缓存架构（Caffeine + Redis）
   一级缓存（本地内存）：使用 Caffeine。当业务系统（如告警系统）拦截到查询请求时，先看自己 JVM 内存里有没有这个用户的权限上下文。有就直接用，耗时接近 0ms。

   二级缓存（分布式缓存）：Caffeine 没命中，再去读 Redis。
   回源持久层：Redis 也没有，再调用权限中心 RPC，查 PostgreSQL 并计算打平，然后回写到 Redis 和 Caffeine。

2. 异步事件失效机制（解决数据不一致）
   本地内存最大的问题是更新不及时。如果管理员在权限中台修改了某个用户的资产组范围，中台已经更新了 PostgreSQL 和 Redis，但业务系统的 JVM 内存里还是老权限，这就导致了权限滞后。

###  针对“阶段三（Rust生产版）”的演进考量

由于需要将这一套逻辑用 Rust 重写，我们在 Java 阶段做 SDK 设计时，就引入了两个非常有利于迁移的伏笔：

逻辑与框架解耦：
我们将“SQL 解析、条件追加、AST 逆向导出”的核心算法全部浓缩在了一个独立的工具类（如 SqlRewriteEngine）中，不与 MyBatis 强绑定。到了 Rust 阶段，你可以无缝将这一套纯逻辑“直译”为基于 sqlparser-rs 的独立函数。

无状态设计：
SDK 内部是完全无状态的，它所需要的上下文参数（租户、资产范围等）完全依赖于入参传入。这完美契合了 Rust 严苛的内存所有权（Ownership）机制，能确保未来在 Rust 的 axum / actix-web 中作为高并发 Middleware（中间件）运行时，具备超高的性能和零内存毛刺。
