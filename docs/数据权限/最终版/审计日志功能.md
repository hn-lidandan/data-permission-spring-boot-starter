## 1. 文档说明

由于系统未来需要支撑 Java MVP + Doris 海量日志检索 + Rust 生产版演进，审计不仅要满足安全合规，更要解决高并发下的吞吐瓶颈和多语言执行流的统一标准化。

## 2. 统一审计日志专项设计方案

### 2.1 审计日志核心流向图

为了确保主查询链路的绝对低延迟（接近零损耗），SDK 内部采用“本地环形缓冲区异步打包 + 消息队列削峰 + 分层落库”的架构。


```
[ 业务服务进程 (Java / Rust SDK) ]
  └── 1. 触发拦截 (Query / Detail / Export / Deny)
  └── 2. 抓取快照并封装 AuditEvent
  └── 3. 投递至本地 RingBuffer (Java: Disruptor / Rust: crossbeam-channel)
         └── 4. 异步批量发送线程 (Batch Poller)
                 │
                 v (异步非阻塞)
     +───────────────────────+
     |   Kafka Broker 集群    |  Topic: data-permission-audit-topic
     +───────────────────────+
                 │
                 v (分区并发消费)
[ 权限服务集群 (Audit Consumer) ]
  └── 5. 实时解析与多级分流策略
         └──  常规审计日志     -> 批量攒批 (1s 或 2000条)
                 │
                 v (高效追加写)
     +──────────────────────────────────────────+
     |             数据存储层                   |
     |  - PostgreSQL (控制面/小流量/事务审计)    |
     |  - Doris (海量检索/数据明细/数据看板)     |
     +──────────────────────────────────────────+

```

### 2.2 数据库设计优化与分层存储策略

 由于日志检索场景（如 Doris 查询）请求量极大，如果所有审计日志都盲目写入 PostgreSQL，数据库性能在生产环境会迅速崩溃。因此实施分层存储。

### 2.2.1 存储分流矩阵

  PostgreSQL：仅存储 POLICY_CHANGE（策略变更）、EXPORT（导出）、DENIED（权限阻断）这类低频、高危、强合规行为。

  Doris：存储 QUERY（列表）、DETAIL（详情）等高频、海量的常规查询快照。 

### 2.2.2 Doris 审计明细表定义（针对海量高性能写入优化）
  在 Doris 中使用冷热分层与 Aggregate/Duplicate 模型，并冗余高基数字段，建立合理的动态分区：

```
CREATE TABLE IF NOT EXISTS dwd_data_permission_audit_di (
    `created_at` DATETIME NOT NULL COMMENT "审计生成时间",
    `tenant_id` VARCHAR(64) NOT NULL COMMENT "租户ID",
    `user_id` VARCHAR(64) NOT NULL COMMENT "操作员用户ID",
    `action` VARCHAR(32) NOT NULL COMMENT "QUERY/DETAIL/EXPORT/DENIED/POLICY_CHANGE",
    `resource_type` VARCHAR(64) NOT NULL COMMENT "资源对象类型",
    
    -- 核心上下文索引
    `client_ip` VARCHAR(64) COMMENT "客户端IP",
    `result` VARCHAR(16) NOT NULL COMMENT "ALLOWED/DENIED",
    `trace_id` VARCHAR(64) COMMENT "分布式链路ID",
    
    -- 文本大字段（非索引列）
    `filter_sql` TEXT COMMENT "SDK 最终重写注入的参数化 SQL 判定条件",
    `original_sql` TEXT COMMENT "业务原始 SQL 摘要",
    
    -- 度量值
    `estimated_rows` BIGINT DEFAULT "0" COMMENT "预估/受影响数据条数",
    `rows_affected` BIGINT DEFAULT "0" COMMENT "实际返回/导出条数",
    `cost_ms` INT DEFAULT "0" COMMENT "权限拦截及SQL改写本身消耗的毫秒数"
) ENGINE=OLAP
DUPLICATE KEY(`created_at`, `tenant_id`, `user_id`, `action`)
COMMENT "数据权限明细审计表"
PARTITION BY RANGE(`created_at`) () -- 依赖动态分区机制
DISTRIBUTED BY HASH(`user_id`) BUCKETS 16
PROPERTIES (
    "dynamic_partition.enable" = "true",
    "dynamic_partition.time_unit" = "DAY",
    "dynamic_partition.start" = "-30",  -- 自动清理 30 天前的数据
    "dynamic_partition.end" = "3",
    "dynamic_partition.prefix" = "p",
    "replication_num" = "3"
);

```

### 2.4 权限服务消费端（Consumer）批处理落库设计

  权限服务作为 Kafka 消费者，如果单条解析并写入数据库，同样会使数据库连接池耗尽。必须引入时间 + 窗口大小双重控制的批量落库机制。

### 2.4.1 攒批消费逻辑

  触发条件：当队列中积压的审计日志达到 2000 条，或者距离上一次提交超过 1 秒（1000ms）。

  执行动作：触发 Doris / PostgreSQL 的 BATCH INSERT。

  对于 Doris，消费端可以利用 Stream Load 接口，以轻量级文本/JSON 流的方式批量推送，这是 Doris 官方推荐的最抗压的导入方案。

### 2.5 针对 Rust 生产版的平滑演进预留
  

在第三阶段演进到 Rust 生产版时，为了保证两套系统审计对齐：

   统一 Kafka JSON Schema：规定多语言 SDK 投递的 JSON 字段严格遵守 11.4.2 审计事件 中定义的标准 Key（全小写蛇形命名）。

   技术组件对齐：Rust SDK 侧采用 tokio::sync::mpsc 或 crossbeam-channel 作为其内部的无锁本地环形缓冲区，替换 Java 的 Disruptor。

   零重构服务端：因为 Kafka 协议、Json Schema、Doris/PG 表结构完全没有变，Rust 业务服务上线时，权限中心的审计消费端不需要改动任何一行代码，即可平滑接入。

### 2.6 核心优势与风险控制
   性能保障：由于引入了无锁本地队列（Disruptor）和 Kafka 异步解耦，引入审计功能后，对业务服务查询的性能损耗控制在 2ms 以内。

   防爆控流机制：万一 Kafka 发生全线崩溃，SDK 本地队列跑满后，启动丢弃策略（Drop Policy）——优先丢弃常规的 QUERY 审计日志，并在本地日志（Logback/Env_log）中打印 Warning，绝对不能因为审计组件挂掉而导致业务不可用（符合 3.1 核心原则 - 业务不硬编码、平台高可用 的本意）。
